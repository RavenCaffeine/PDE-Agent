# AI4S CNS PDE Agent 赛题解读

## 1. 赛题本质

本赛题不是传统的“提交一个最优神经网络权重”比赛，而是评测一个面向 PDE 神经算子的科研智能体。评委希望看到 Agent 是否能自动完成：

1. 理解 PDEBench 数据、基线模型与物理问题。
2. 诊断模型在短期精度、长时稳定性、泛化能力上的瓶颈。
3. 修改代码并设计实验，而不是只调用现成模型。
4. 运行训练、推理、验证，并把完整科研轨迹写入 JSONL 日志。

因此，最终提交物包含两层价值：

- 预测文件：`task{N}_pred.hdf5`，直接决定精度得分。
- 科研过程证据：`task{N}_logs.log`、`methodology.pdf`、`code/`，决定是否合规以及能否体现 Agent 能力。

## 2. 初赛任务拆解

### Task 1：固定物理环境 Burgers 短窗预测

- 数据：`1D_Burgers_Sols_Nu0.001.hdf5`，原始维度约为 10000 个样本、200 时间步、1024 空间点。
- 官方模型训练时做了降采样：时间 `200 / 5 = 40`，空间 `1024 / 4 = 256`。
- 新验证集扩展为 1000 时间步、1024 空间点，评测时相当于 200 时间步、256 空间点。
- 测试输入给前 10 个时间步，要求预测完整 `(N, 200, 256)`，其中前 10 步必须与输入一致。
- 可加载官方 checkpoint 微调。

核心挑战：

- 单步误差会在自回归预测中累积。
- 评分包含短期 Rel-MSE 和中长期统计分布匹配。
- 训练耗时和推理耗时都计分，Task 1 不能只追求大模型。

### Task 2：多物理环境 Burgers 泛化

- 训练集包含多个粘性系数 `Nu`，每个样本提供 `Nu`。
- 测试集不提供 `Nu`，只能基于初始条件预测。
- 不能使用任何公开预训练权重，必须从本题训练集从头训练。
- 只评测预测精度，但推理必须在 2 分钟内，总时长需控制在 12 小时内。

核心挑战：

- 训练阶段可利用 `Nu`，推理阶段不能显式依赖 `Nu`。
- 需要从前 10 步中隐式识别物理参数，或训练一个参数推断分支。
- 多物理环境下模型若只记忆平均动力学，长时预测会迅速漂移。

### Task 3：KS 方程附加题

- 输入：每个样本前 20 步，shape `(20, 256)`。
- 输出：完整 400 步，shape `(400, 256)`，前 20 步必须复制输入。
- 训练/验证提供 `lambda2`，测试不提供。
- KS 是混沌系统，长期逐点预测天然困难，因此第三段评分更看重统计分布。

核心挑战：

- 前 20 步要同时承担状态识别和参数识别。
- 长时点态误差不可避免，需要优化统计一致性、谱能量、均值方差、自相关等物理统计量。
- 训练策略应兼顾短期 teacher forcing、长窗 rollout、频谱正则和稳定性约束。

## 3. 评分公式对策略的影响

Task 1/2 只评估预测的 190 步：

- 第 1 段：前 47 步，权重 25%，`100 * exp(-20 * Rel-MSE)`。
- 第 2 段：47-95 步，权重 25%，`100 * exp(-10 * Rel-MSE)`。
- 第 3 段：95-190 步，权重 50%，取 Lorentzian 与 Fréchet 统计分数的较大值。

这意味着：

- 前两段必须尽量降低逐点误差。
- 第三段不一定要逐点完全正确，但必须维持合理统计分布。
- 损失函数不能只用全窗 MSE，应该分段加权，并加入频谱/统计正则。

Task 3 类似，但预测区间为 380 步：

- 第 1 段：步 20-49，强调短期逐点准确。
- 第 2 段：步 50-199，强调中期轨迹。
- 第 3 段：步 200-399，强调统计分布。

## 4. 合规红线

必须避免：

- 调用数值求解器生成测试预测。
- Task 2/3 使用公开预训练权重。
- 缺失日志或日志不是逐行 JSON。
- `code/` 中代码无法从 Agent 日志中追溯。
- 推理超过 2 分钟。
- 提交文件前 10/20 步没有严格复制测试输入。

允许：

- Task 1 使用官方 PDEBench checkpoint 微调。
- 训练阶段使用训练集提供的 `Nu` 或 `lambda2`。
- 引入物理损失、频谱损失、稳定性损失。
- 设计 Agent 自动读日志、修改代码、运行实验。

## 5. 比赛交付物理解

推荐提交目录：

```text
submission/
├── submission.json
├── task1_pred.hdf5
├── task1_time.csv
├── task1_logs.log
├── task2_pred.hdf5
├── task2_time.csv
├── task2_logs.log
├── task3_pred.hdf5
├── task3_time.csv
├── task3_logs.log
├── methodology.pdf
└── code/
```

每个任务至少需要：

- `task{N}_pred.hdf5`
- `task{N}_time.csv`
- `task{N}_logs.log`

其中 `logs.log` 是赛事核心材料之一，需要逐行 JSON，并至少包含：

- `timestamp`
- `elapsed_seconds`
- `response` 或 `tool_calls`

## 6. 优先级建议

第一优先级：Task 1。

- 规则允许微调官方 checkpoint。
- 物理环境固定，最容易建立可靠 MVP。
- Task 1 同时影响方案 A 和方案 B。

第二优先级：Task 3。

- 附加题在方案 B 中可放大到 200 分。
- 混沌系统长时统计评分给了方法创新空间。

第三优先级：Task 2。

- 需要从头训练，且测试不提供 `Nu`。
- 若训练资源有限，Task 2 可作为方案 A 的后续增强。
